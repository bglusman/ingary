Draft Product Architecture

Ingary Extraction

A standalone product for stable virtual model contracts, explainable routing, bounded stream governance, and receipts.

Working product name: Ingary. Easy to rename later, but using one name keeps the architecture and prototype work concrete.

The Thesis

What users call

  • coding-balanced
  • local-first-private
  • long-context-safe
  • json-extractor-cheap
  • premium-review

What the product owns

  • route graph and provider selection
  • context-window fit checks
  • fallback and escalation policy
  • stream governance before release
  • receipts, simulation, evals, rollout state
The product should not compete as "another LLM gateway." It should make model behavior a versioned, explainable contract.

Where Calciforge Leaves Off

Calciforge keeps
  • channel routing
  • agent identity and commands
  • secret substitution
  • security proxy / host-agent boundaries
New product owns
  • synthetic model graph engine
  • provider adapter abstraction
  • OpenAI-compatible serving API
  • stream policy and receipts
  • model behavior UI
Caller traceability is best effort: trust server-side auth first, then explicit headers or request metadata, then provider/gateway records, then stable anonymous caller grouping.

Latest Calciforge main already generalized OpenAI-compatible provider adapters and added streaming/telemetry primitives, which makes the extraction boundary clearer.

Deployment Topologies

In front of LiteLLM / Helicone Clients call synthetic models directly. Best receipts and stream governance. Must forward caller metadata downstream.
Behind an existing gateway Register Ingary as one OpenAI-compatible backend. Use ingary/coding-balanced if the gateway has a shared model namespace.
Embedded library Host app or Calciforge uses the Rust planner/governor directly. Low overhead, but UI needs receipt sync.
Standalone direct gateway Simple local/small deployment. Strong end-to-end receipts, but product owns more provider behavior.
Recommended default: expose synthetic models as the public unit. Prefix with ingary/ only when another gateway owns a larger shared model namespace.

Shareable Model Artifacts

Portable manifests Route graphs, stream rules, capability requirements, eval references, compatibility notes, author/license/provenance.
Provider-role mapping Artifacts define abstract roles like small_local_coder; operators map them to local concrete models.
Hub workflow Browse, import as draft, simulate, run evals, activate, and track updates.
Trust boundary No credentials, private endpoints, or real captured user content in exported artifacts by default.
A community hub is optional infrastructure. Local file artifacts and private Git repositories should work first.

Build Sequence

MVP 1: Headless Core route graph, validation, simulation, receipt schema, token estimators
MVP 2: Standalone Gateway OpenAI-compatible chat, model list, SQLite receipts, mock/OpenAI HTTP adapter
MVP 3: Stream Governance normalized events, buffered/full modes, declarative regex rules, retry/escalate
MVP 4: UI catalog, providers, graph view, simulator, receipts, stream policy editor
Programmable policy should follow the declarative event model, not precede it. Otherwise the product risks becoming a policy VM before the core behavior is understandable.

Open Decisions

  • Name is Ingary for now; revisit only after validating the product shape.
  • Start in a new repository immediately or stabilize interfaces in Calciforge first.
  • Calciforge integration path: Rust API, HTTP boundary, or hybrid.
  • How much provider-adapter config should migrate directly from Calciforge.
  • Whether weighted alloys belong in the first public MVP.
  • Minimum UI that proves differentiation: simulator + receipts may matter more than graph editing.
The strongest proof point is a receipt that explains a route decision and a stream-policy retry that prevented bad output from reaching an agent.
